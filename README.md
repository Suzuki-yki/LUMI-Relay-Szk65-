# LUMI Relay

> 一个让任何人通过“一次只改一个细节”的方式，共同创作并接力演化 LUMI 角色的链上社交游戏。

LUMI Relay 是部署在 Monad 上的低门槛角色共创产品。用户不需要先理解 NFT、钱包或 Web3；他们可以先像玩换装游戏一样参与创作，再选择将作品发布到链上，留下可验证的创作来源与归属记录。

## 项目愿景

LUMI 的长期目标是构建一个包含 AI 陪伴、数字房间、角色、装备与 Marketplace 的数字世界。

LUMI Relay 是这个大项目的“社区共创入口”：每个人都可以接过一只已有的 LUMI，只改变一个外观元素，再把它传给下一位用户。随着参与者增加，社区会共同形成一棵不断分叉、可追溯的 LUMI 角色进化树。

```text
官方基础 LUMI
├── 蓝裙 LUMI
│   ├── 猫耳 LUMI
│   │   └── 雨天 LUMI
│   └── 红发 LUMI
└── 运动装 LUMI
    └── 朋克配饰 LUMI
```

## 用户问题

许多人认为 Web3 与 NFT 复杂、金融化，并且离日常兴趣很远。用户通常需要先了解钱包、Gas、交易和 Token，才能开始体验产品；而不少 NFT 产品又只强调“拥有”，缺少持续的创作和社交互动。

LUMI Relay 希望提供一种更自然的体验：

- 不需要绘画能力；
- 不需要 Web3 知识即可开始；
- 每位用户只需完成一个小创作动作；
- 每一步创作都有可验证的来源关系；
- 社区共同创造一个持续成长的角色世界。

## 核心玩法

1. 用户无需连接钱包，直接打开 LUMI Relay。
2. 用户选择一只由其他人创作过的 LUMI。
3. 用户点击“接力这只 LUMI”。
4. 用户只能修改一个类别：发型、服装、配饰或配色。
5. 用户为新角色写下一句简短故事。
6. 页面生成新的角色预览卡，用户可以立即分享。
7. 用户选择发布时，连接钱包并将 Remix 发布到 Monad。
8. 公开画廊展示新角色、父角色、创作者和所在分支。

## MVP 范围

本项目先完成一个小而完整、可运行、可展示、可验证的原型。

### 本周真实实现

- 一只官方基础 LUMI 角色；
- 四类可组合素材：发型、服装、配饰、配色；
- 每类 3–4 个初始选项；
- 角色实时预览；
- 每次 Remix 只能修改父角色的一个类别；
- 角色故事输入；
- 钱包连接；
- Monad 测试网发布；
- 简单的公开 Remix 画廊与父子关系展示。

### 暂不实现

- 自由上传任意图片；
- AI 图片生成或自动审核；
- Marketplace；
- Token 经济模型；
- DAO 治理与投票；
- 复杂 3D 模型；
- 嵌入式钱包与社交登录。

## 为什么使用区块链

LUMI Relay 不是把图片简单标记为 NFT。链上数据承担的是创作谱系记录：

- 该角色来自哪一只父角色；
- 谁发布了这一版 Remix；
- 使用了哪些外观素材；
- 何时创建；
- 相同的素材组合是否已经被发布。

因此，每个角色都是社区共同创作历史中的一个节点，而不是一张孤立图片。

## 智能合约设计

MVP 使用 ERC-721 作为角色 NFT 标准。

每个角色记录：

- `parentTokenId`：父角色 ID；
- `hairId`：发型素材 ID；
- `outfitId`：服装素材 ID；
- `accessoryId`：配饰素材 ID；
- `paletteId`：配色素材 ID；
- `creatorStory`：创作者留下的角色故事；
- `traitHash`：由素材组合生成的唯一哈希。

核心函数：

```solidity
createBaseLumi(...)

createRemix(
    uint256 parentTokenId,
    uint8 hairId,
    uint8 outfitId,
    uint8 accessoryId,
    uint8 paletteId,
    string creatorStory,
    string metadataURI
)
```

合约需要验证：

- 父角色存在；
- 所有素材 ID 均在允许范围；
- 新角色只修改父角色的一个外观类别；
- 相同组合不能重复发布；
- 新角色正确记录父角色 ID。

成功创建后，合约发出 `RemixCreated` 事件，前端据此建立 Remix Tree。

## 内容安全策略

智能合约无法直接理解图片，因此不能可靠地判断上传的图片是否违规、是否符合 LUMI 模型，或是否与官方素材重复。

MVP 采用“结构性限制”保证安全：用户只能组合官方提供的素材；前端不开放任意图片上传；合约只接受允许的素材 ID，并通过 `traitHash` 阻止重复组合。

未来允许用户上传自制素材时，流程将升级为：

1. 用户在链下上传素材；
2. AI 检查素材是否符合 LUMI 模板与内容规范；
3. 人工或社区审核确认；
4. 系统为通过的素材生成审核签名；
5. 合约验证签名后，允许素材进入 LUMI 素材库。

AI 负责链下识别与审核，智能合约负责链上验证审核结果。

## 技术栈

- 智能合约：Solidity、OpenZeppelin ERC-721
- 开发环境：Hardhat
- 区块链网络：Monad 测试网
- 前端：React、TypeScript、Tailwind CSS
- 钱包交互：wagmi、viem
- 数据与元数据：MVP 使用本地数据或 Supabase，后续可接入 IPFS
- 代码管理：GitHub

Monad 兼容 EVM，可使用 Solidity、OpenZeppelin 和 Hardhat 等现有工具，同时适合构建需要快速响应和多人参与的链上互动产品。

## AI 辅助开发

AI 作为开发助手，而不是替代开发判断：

| 开发阶段 | AI 的作用 | 开发者负责内容 |
| --- | --- | --- |
| 产品规划 | 拆解需求与 MVP | 决定真实问题和范围 |
| 合约设计 | 建议数据结构、边界情况 | 判断业务规则与安全性 |
| 前端开发 | 生成组件骨架、辅助调试 | 整合、测试与调整代码 |
| 合约测试 | 提出测试场景 | 实际执行并验证结果 |
| 文档整理 | 协助 README、架构说明和 Demo 文案 | 确保内容与项目一致 |

- 用户无需理解 Web3，即可完成角色创作与预览；
- 用户连接钱包后，可在 Monad 测试网发布 Remix；
- 合约正确记录父角色与新角色的关系；
- 重复素材组合会被拒绝；

- 项目有公开 GitHub Repo、合约地址、交易哈希和演示视频。
